Brooklyn Nine Nine

My notes for the Brooklyn Nine Nine room. This is a simple walkthrough of how I moved from anonymous FTP to SSH access, then used a sudo rule to read the final flag.

Fast Linux FTP SSH Hydra nopasswd
Small note before reading: I do not usually write write-ups, so this is more of a clean learning log than a polished guide. Also, if you are reading this just to copy answers, you are using it the wrong way. A write-up should be used when you are genuinely stuck or when you want to understand the reasoning behind a step.

0. Plan

The goal was not to run random commands until something worked. The idea was: identify services, look for information leaks, use that information to get a valid login, then check what the user can access.

1. Initial scan

I started with Nmap using default scripts and version detection. This gives a quick view of the exposed services and sometimes already reveals useful misconfigurations. 

nmap -sS -sV -sC TARGET_IP

The useful result was that FTP and SSH were open. The important detail was not just that FTP existed, but that anonymous login was allowed.

2. Anonymous FTP

Since anonymous FTP was enabled, I connected without real credentials. This is worth checking because public FTP access often exposes notes, backups, usernames or other hints. 

ftp TARGET_IP
Name: anonymous

After logging in, I listed the directory and found one file: note_to_jake.txt. I downloaded it locally to read it. 

ls
get note_to_jake.txt

3. Reading the note

The file was the main clue. It mentioned jake and said his password was weak. That gives two things at once: a likely username and a reason to test SSH authentication. 

cat note_to_jake.txt

4. Testing SSH

SSH was open from the scan, and now I had a username. I used Hydra with RockYou to test the weak-password hint from the note. 

hydra -l jake -P /usr/share/wordlists/rockyou.txt TARGET_IP ssh

Hydra found a valid password for jake. At this point, the FTP information disclosure turned into real shell access.

5. User flag

After logging in over SSH, I checked the accessible home directories and read the user flag. In this room, the flag was not in Jake's own home directory; it was inside /home/holt. 

ssh jake@TARGET_IP
cd /home/holt
cat user.txt

6. Checking sudo

After getting a shell, one of the first checks is sudo -l. It shows what the current user is allowed to run with elevated privileges. 

sudo -l

The result showed that jake could run /usr/bin/less as root without a password. For this room, I did not need to spawn a root shell. I only needed to read the final flag.

7. Root flag

Because less was allowed through sudo, I used it directly to open /root/root.txt. 

sudo less /root/root.txt

That opened the file and completed the machine.
